# CLAUDE.md

## Project context

A CRUD application for employee management. Users (`User`) authenticate and manage (create, read, update, delete) employee (`Employee`) records.

## Stack

- **.NET 10**
- **ASP.NET Core MVC** (Controllers, Models, Views — no Minimal APIs, no Razor Pages)
- **Entity Framework Core** (Code First, migrations)
- **SQL Server**
- **Authentication**: Native ASP.NET Core cookie-based auth (`AddAuthentication().AddCookie()`). **Do not use full ASP.NET Core Identity.**
- **Password hashing**: `Microsoft.AspNetCore.Identity.PasswordHasher<T>` (namespace only, not the full Identity system)

## Explicit rules — what NOT to do

- Do not use full ASP.NET Core Identity (`UserManager`, `SignInManager`, `AspNetUsers` tables, etc.)
- Do not use Minimal APIs
- Do not implement password encryption/decryption. Passwords are **hashed** (`PasswordHasher<T>.HashPassword`) and **verified** (`VerifyHashedPassword`) — never decrypted.
- Do not put business logic directly in Controllers. Controllers call Services.
- Do not manually edit files generated in `Migrations/`. Always regenerate with `dotnet ef migrations add`.
- Do not perform physical `DELETE` operations on `User` and `Employee` table's records. All deletion is a soft-delete via `IsActive = false`.
- Do not write queries that bypass the global `IsActive` filter except with an explicit `.IgnoreQueryFilters()`, and only when the use case justifies it (e.g. auditing).

## Data schema

### `User` table
The entity that authenticates into the system and creates employees.

| Column      | Type                           | Notes                                   |
|-------------|---------------------------------|------------------------------------------|
| Id          | PK                              |                                          |
| Email       | string                          | used as login/username                  |
| Password    | string                          | hash generated by `PasswordHasher<T>`   |
| FirstName   | string                          |                                          |
| LastName    | string                          |                                          |
| Role        | enum (`Admin`, `Standard`)      | mapped with `HasConversion<string>()`   |
| CreatedAt   | DateTime                        | automatic audit field                   |
| ModifiedAt  | DateTime                        | automatic audit field                   |
| IsActive    | bool                             | soft-delete, global query filter        |

### `Employee` table

| Column          | Type                           | Notes                                           |
|-----------------|----------------------------------|---------------------------------------------------|
| Id              | PK                               |                                                    |
| FirstName       | string                           |                                                    |
| LastName        | string                           |                                                    |
| Type            | enum (`Manager`, `General`)      | mapped with `HasConversion<string>()`             |
| CreatedByUserId | FK -> `User.Id`                  | required, `OnDelete(DeleteBehavior.Restrict)`     |
| CreatedAt       | DateTime                         | automatic audit field                             |
| ModifiedAt      | DateTime                         | automatic audit field                             |
| IsActive        | bool                              | soft-delete, global query filter                  |

**Note**: since deletion is always logical (`IsActive`), a `User` never physically disappears from the table, so any `Employee` records referencing it remain intact. `DeleteBehavior.Restrict` is an additional safeguard against an accidental physical `DELETE`, not the primary integrity mechanism.

## Implementation patterns

### Enums mapped as strings
```csharp
public enum UserRole { Admin, Standard }
public enum EmployeeType { Manager, General }
```
In `OnModelCreating`:
```csharp
modelBuilder.Entity<User>()
    .Property(u => u.Role)
    .HasConversion<string>();

modelBuilder.Entity<Employee>()
    .Property(e => e.Type)
    .HasConversion<string>();
```

### Automatic auditing (`CreatedAt` / `ModifiedAt`)
`IAuditable` interface implemented by `User` and `Employee`. Override `SaveChangesAsync` in the `DbContext`:
```csharp
public override async Task<int> SaveChangesAsync(CancellationToken cancellationToken = default)
{
    var entries = ChangeTracker.Entries()
        .Where(e => e.Entity is IAuditable &&
                    (e.State == EntityState.Added || e.State == EntityState.Modified));

    foreach (var entry in entries)
    {
        var entity = (IAuditable)entry.Entity;
        if (entry.State == EntityState.Added)
            entity.CreatedAt = DateTime.UtcNow;
        entity.ModifiedAt = DateTime.UtcNow;
    }

    return await base.SaveChangesAsync(cancellationToken);
}
```
Any new entity requiring auditing should implement `IAuditable`; do not hardcode concrete types in the override.

### Global soft-delete (`IsActive`)
Global query filter in `OnModelCreating`:
```csharp
modelBuilder.Entity<User>().HasQueryFilter(u => u.IsActive);
modelBuilder.Entity<Employee>().HasQueryFilter(e => e.IsActive);
```
"Deleting" a record means updating `IsActive = false` via a Service, never calling `Remove()`.

### `Employee.CreatedByUserId` relationship
```csharp
modelBuilder.Entity<Employee>()
    .HasOne<User>()
    .WithMany()
    .HasForeignKey(e => e.CreatedByUserId)
    .OnDelete(DeleteBehavior.Restrict);
```

### Cookie-based authentication
- Configured in `Program.cs` via `AddAuthentication(CookieAuthenticationDefaults.AuthenticationScheme).AddCookie(...)`
- Cookie settings:
  - `Cookie.Name` → explicit app-specific name (never leave the `.AspNetCore.Cookies` default)
  - `Cookie.HttpOnly = true`
  - `Cookie.SecurePolicy = CookieSecurePolicy.Always` (HTTPS only)
  - `Cookie.SameSite = SameSiteMode.Lax`
- Login: validate `Email` against `User`, verify `Password` with `PasswordHasher<User>.VerifyHashedPassword`
- Minimum claims when calling `SignInAsync`:
  - `ClaimTypes.NameIdentifier` → `User.Id` (used as `CreatedByUserId` when creating employees)
  - `ClaimTypes.Name` → `User.Email`
  - `ClaimTypes.Role` → `User.Role`
- Authorization via `[Authorize]` at controller/action level; `[Authorize(Roles = "Admin")]` for restricted functionality
- Logout via `SignOutAsync`

## Verification commands

Claude Code should run these commands to verify its own work after relevant changes:
- `dotnet build` — after touching Controllers, Services, or Models
- `dotnet ef migrations add <DescriptiveName>` — after schema changes to entities
- `dotnet ef database update` — only when explicitly requested to apply migrations

Destructive commands (`dotnet ef database drop`, any direct `DELETE` against SQL Server) require explicit user confirmation before execution.
